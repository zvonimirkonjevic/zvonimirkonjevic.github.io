Performance isn't a feature — it's a *requirement*. Users expect pages to load in under 2 seconds, and search engines reward fast sites with better rankings. Let's talk about what actually matters.

## The Core Web Vitals

Google's Core Web Vitals are three metrics that capture the *user experience* of loading a page:

| Metric | Measures | Good Threshold |
|--------|----------|----------------|
| **LCP** (Largest Contentful Paint) | Loading speed | ≤ 2.5s |
| **INP** (Interaction to Next Paint) | Responsiveness | ≤ 200ms |
| **CLS** (Cumulative Layout Shift) | Visual stability | ≤ 0.1 |

## Optimizing LCP

The Largest Contentful Paint measures when the *biggest* piece of content becomes visible. Usually this is a hero image or heading.

### Quick Wins

1. **Preload critical resources** — Use `<link rel="preload">` for hero images
2. **Optimize fonts** — Use `font-display: swap` and subset your fonts
3. **Server-side render** — Or better yet, statically generate your pages

```html
<!-- Preload your hero image -->
<link rel="preload" as="image" href="/hero.webp" />

<!-- Preload critical font -->
<link
  rel="preload"
  as="font"
  type="font/woff2"
  href="/fonts/inter.woff2"
  crossorigin
/>
```

## Taming CLS

Layout shifts are *infuriating* for users. You're about to tap a button and suddenly everything moves. The fix is simple in principle:

> Reserve space for every element before it loads.

```css
/* Always set explicit dimensions for images */
img {
  width: 100%;
  height: auto;
  aspect-ratio: 16 / 9;
}

/* Reserve space for ads or dynamic content */
.ad-slot {
  min-height: 250px;
}
```

## The Performance Budget

I recommend setting a performance budget and sticking to it:

- **JavaScript bundle**: ≤ 150KB (gzipped)
- **Total page weight**: ≤ 1MB
- **HTTP requests**: ≤ 30
- **Time to Interactive**: ≤ 3s on 3G

Tools like `bundlesize` and Lighthouse CI can enforce these automatically in your CI/CD pipeline.

## Measuring in Production

Lab data (Lighthouse) is useful but **field data** is truth. Use the Web Vitals library to capture real user metrics:

```typescript
import { onLCP, onINP, onCLS } from "web-vitals";

onLCP(console.log);
onINP(console.log);
onCLS(console.log);
```

Then send these metrics to your analytics dashboard and track them over time. Performance is a *continuous* effort, not a one-time optimization.

## Key Takeaways

1. **Measure first** — You can't improve what you don't measure
2. **Focus on user impact** — Core Web Vitals capture what users actually feel
3. **Set budgets** — Automated enforcement prevents regression
4. **Monitor in production** — Lab data ≠ real-world performance
